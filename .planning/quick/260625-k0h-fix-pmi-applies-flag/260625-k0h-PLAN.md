<objective>
Fix the `pmiDropOffMonth = null` ambiguity in the TCO engine. Today `null` conflates TWO
states — "PMI does not apply" and "PMI applies but the amortization schedule never reaches the
drop-off LTV threshold within the term." `buyMonthlyOutflowAt` (rent-vs-buy.ts:129) gates PMI on
`pmiDropOffMonth !== null && month <= pmiDropOffMonth`, so the second case silently charges $0 PMI
for the entire hold — biasing the rent-vs-buy verdict toward BUY — while `tco.pmi.annualized`
correctly charges it (via `?? totalMonthsHeld`). The two surfaces disagree.

Fix: add an explicit `pmiApplies: boolean` to `TcoBreakdown`, populate it from `pmiResult.applies`,
and re-gate `buyMonthlyOutflowAt` to charge PMI whenever `pmiApplies && (pmiDropOffMonth === null ||
month <= pmiDropOffMonth)`. This closes CR-01 from `02-REVIEW-gap-closure.md`.

Purpose: the rent-vs-buy verdict is the product's core value ("don't buy / rent and invest the
difference"). A silent BUY bias corrupts the headline answer.
Output: updated `TcoBreakdown` shape, corrected gating, a RED-first locking test, and a regenerated
golden snapshot (additive `pmiApplies` key only — explained below).
</objective>

## CRITICAL PRE-WORK FINDING — read before writing the RED test

The planner empirically verified (built `packages/core/dist`, swept down-payment × term × rate):
**through the standard `computeTco` path, `pmiResult.applies === true && pmiResult.dropOffMonth ===
null` is UNREACHABLE.** The amortization schedule always ends with a reconciled `$0.00` balance row
(amortization.ts:114-125), and `$0 <= 78%` of the original value, so `computePmi` always finds a
finite drop-off month (worst case = `termMonths`). The "short term + LTV just over 80%" idea in the
task description does NOT produce a null drop-off — a short term amortizes FAST, so drop-off lands
EARLY.

Consequence for the test strategy: you CANNOT trigger the defect end-to-end through `rentVsBuy`
with real scenario inputs. The bug is a **latent / defensive correctness defect** in the gating
predicate — the two PMI surfaces disagree *in principle* (and would diverge for any future
non-$0-terminating schedule the schema permits). So the RED test must target the **gating predicate
at its seam**, not a full `rentVsBuy` run.

Required approach: extract the PMI charge decision into a small PURE, EXPORTED predicate
`shouldChargePmi(pmiApplies: boolean, pmiDropOffMonth: number | null, month: number): boolean` in
`rent-vs-buy.ts`, and have `buyMonthlyOutflowAt` call it. The RED test then asserts
`shouldChargePmi(true, null, 5) === true` (the applies-true / drop-off-null case). This FAILS before
the fix (the symbol does not exist / old inline logic returns false) and PASSES after. This is the
honest, reachable reproduction of CR-01. Do NOT fabricate a `rentVsBuy` scenario that "never drops
off" — there is none.
