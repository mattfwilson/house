<objective>
Define the SQLite schema as Drizzle tables (`profiles`, `scenarios`), wire the better-sqlite3 + Drizzle connection + programmatic migrator (`db.ts`), then GENERATE and commit the real migration SQL and prove it applies to a live database. Money columns are TEXT (canonical decimal strings), the scenario `snapshot` is a TEXT blob (the canonicalJson frozen-snapshot source of truth, D-05), and a unique index enforces scenario-name-uniqueness-within-profile (D-11).

**CRITICAL (refreshed research):** the `profiles` table persists the FULL NINE Household leaves (`engine-input.ts:116-154`), NOT the five PROF-01-visible fields. PROF-01 "net worth" maps to the column `available_net_worth`. Do NOT add a separate `net_worth` column. **The 06-PATTERNS.md schema sketch (§ schema.ts) is STALE — it shows a `net_worth` column and only ~five leaves; follow the refreshed 06-RESEARCH.md § "Drizzle schema sketch (decimal-string money discipline — NINE profile leaves)" instead.** A profile that persists fewer than nine leaves cannot drive the affordability/FI engines.

Purpose: This is the persistence substrate every repository in 06-05 builds on. The [BLOCKING] migration task is the load-bearing proof that the schema is materialized in a real DB — Drizzle's row types come from the schema module, so a passing `tsc -b` is a FALSE-POSITIVE for "the table exists." Only a migrate-then-insert/select round-trip proves it.
Output: `schema.ts`, `db.ts`, committed `drizzle/*.sql` migration(s), and a migration test that round-trips a real nine-leaf row through a fresh `:memory:` DB.
</objective>
