<objective>
Define the Phase-6 contracts in `packages/core` as pure TypeScript: the repository ports (`ProfileRepository`, `ScenarioRepository`), the `ListingsProvider` port, and the domain types (`Profile`, `Listing`, `SavedScenario`) — each money leaf a canonical decimal string behind a Zod `.strict()` boundary. These are the inward-facing contracts every `packages/app` adapter and service depends on (D-02 dependency inversion).

Two refreshed-research points are load-bearing here:
1. **`Profile = { id, name } & Household`, and `Household` has NINE money/rate leaves** (`engine-input.ts:116-154`), not the five PROF-01-visible fields. A profile that captures only the five visible fields cannot drive the affordability/FI engines. PROF-01 "net worth" maps to the engine field `availableNetWorth` — do NOT invent a separate `netWorth` leaf.
2. **`SavedScenario` embeds a FROZEN copy of the EngineInput snapshot** (`{ asOf, assumptions, scenario, household? }`). The snapshot is self-contained; reload rebuilds the result solely from the embedded blob and NEVER re-joins the live profile row, so a later profile edit can never retroactively change a saved scenario (PROF-04 reproducibility).

Purpose: The side that defines the need owns the contract. Locking these interfaces first lets the Wave-2/3 adapters be written against a known shape instead of exploring the codebase (interface-first ordering). LIST-01 ("clean ListingsProvider interface, isolated from the core") is satisfied the moment these compile and export.
Output: Five new pure files + barrel exports + a no-bare-number type-test. Zero runtime behavior; zero framework deps added to core.
</objective>
