# Phase 5 — UI Design Contract

> Visual and interaction contract for frontend phases. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope Notice (read first)

Phase 5 is a **pure calculation-core phase**. Per `.planning/research/ARCHITECTURE.md` (build steps 8–9) and the ROADMAP, the Town Scoring engine — weighted normalized composite, realistic/stretch/fantasy bucketing, per-metric contributions, and MA-specific flags — lives in `packages/core/src/towns/`. There is **no `apps/web`, no Next.js, no Tailwind, no shadcn** in the repo at this phase, and none is introduced here. The actual heatmap **rendering** (pixels, components, layout, app chrome) is delivered in **Phase 7 (Web Shell)** — the only phase marked `UI hint: yes`.

Therefore this document is a **heatmap visualization contract**, not a conventional app UI-SPEC. It pins the part of the visual/interaction contract that the **Phase-5 engine output must satisfy** so that **Phase 7 can render the heatmap faithfully**: color/encoding semantics, the bucket palette, missing-data treatment, per-metric explainability shape, the MA-flag presentation contract, and the bucket/flag/empty/error copy.

**Deferred to Phase 7 (not specified here):** shadcn init, component library, icon library, font family, the 8-pt spacing scale, app-wide typography roles, page layout, and primary navigation. Those are app-chrome decisions that belong with the shell that consumes this engine.

**Renderer assumption:** the contract is **renderer-agnostic**. Phase 7 is expected to use **visx (@visx/visx 4)** for the dedicated heatmap view (the stack-sanctioned escalation for the town heatmap), but a CSS-grid table-heatmap fallback remains valid — because what is fixed here is the **data + encoding semantics**, not the drawing technology.

## Color

This is the **load-bearing Phase-5 contract.** It defines the semantic meaning each color must carry; Phase 7 must honor these meanings when it renders. The palette is **colorblind-safe** (distinct hue + lightness, not red/green-only) and **anti-funnel**: affordability tiers read as honest disclosures, never as a "buy this town" nudge. No bucket is rendered in success-green or aspirational styling.

### Bucket palette (categorical — the primary encoding)

| Bucket | Hue (base) | Meaning | Anti-funnel note |
|--------|-----------|---------|-----------------|
| Realistic | teal `#0F766E` | median price ≤ entered budget | Not green; teal reads "within reach," not "you should buy" |
| Stretch | amber `#B45309` | budget < median ≤ budget × stretchFactor | Caution tone, honest "above comfort" |
| Fantasy | slate-gray `#64748B` | median above the stretch ceiling | Neutral/muted — out of reach, de-emphasized, never red-as-error |
| No data | hatched gray `#94A3B8` | a contributing metric is missing | Explicit "no data" marker — **never** a silent 0, blank, or imputed value |

### Score intensity (sequential — the secondary encoding, within a bucket)

- The composite score modulates **lightness within the bucket's base hue** (darker = stronger composite within that tier). Hue still encodes the bucket; lightness encodes degree.
- Intensity must be **monotonic and continuous** so a reader can rank towns within a bucket by shade. The engine must expose a normalized composite scalar (0–1) per town for this mapping (see Heatmap Encoding Contract).

### Standard color-role table

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | deferred to Phase 7 | App background, surfaces |
| Secondary (30%) | deferred to Phase 7 | Cards, sidebar, nav |
| Accent (10%) | `#0F766E` (teal, = Realistic) | Reserved for: the Realistic bucket and the active-budget control only |
| Destructive | deferred to Phase 7 | Not used by the heatmap; MA flags are informational, NOT destructive |

**Accent reserved for:** the Realistic bucket fill and the budget-input control that drives bucketing — nothing else. Stretch (amber) and Fantasy (slate) are bucket semantics, not accent. MA flag chips are neutral/informational and never consume the accent or a destructive color.

## Heatmap Encoding Contract

The Phase-5 engine output **must** carry the following so Phase 7 (visx or CSS-grid) can render the heatmap without re-deriving any financial logic. This is the interaction/visual contract proper.

| Contract item | Requirement |
|---------------|-------------|
| Per-town composite | A normalized composite **scalar in [0,1]** plus the raw composite, each metric already scaled and direction-corrected (higher = better) per success criterion 1. |
| Per-metric contribution | For each town, an **itemized, explainable** breakdown: per metric → `{ rawValue, normalizedValue, direction, weight, weightedContribution, missing: boolean }`. The heatmap cell tooltip/expander renders this verbatim — "each town's per-metric contribution explainable" (criterion 3) is satisfied by data, not by UI guesswork. |
| Metrics in scope | mill rate, median price, commute to a **configurable anchor**, school rating, and custom amenity weights. Direction-correction: lower mill rate / lower median price / shorter commute = better; higher school rating / higher amenity match = better. |
| Bucket field | Each town carries a `bucket: 'realistic' \| 'stretch' \| 'fantasy'` enum tied to the entered budget. The renderer maps enum → palette above; it never recomputes the bucket. |
| Missing data | Missing metrics are flagged **explicitly** (`missing: true`, value excluded from the composite per the engine's no-silent-0/NaN rule) so the renderer can draw the hatched-gray "no data" treatment. The renderer must not impute, zero-fill, or hide a missing cell. |
| Commute anchor | Surfaced **generically** as "commute to anchor" in the legend/label — **no hardcoded default town**; the anchor is a configurable input echoed back in the output for display. |
| Matrix shape | Output is renderable as a towns × metrics matrix (rows = towns, columns = contributing metrics) with an additional composite/bucket summary column — directly consumable by visx heatmap scales or a CSS grid. |
| Sort/legend | Output must let the renderer sort towns by bucket then by normalized composite, and render a legend covering all four palette states (incl. "no data"). |

## MA-Specific Flags Contract

Per TOWN-04, MA realities are **qualitative flags**, not dynamic simulations. Each flag is rendered as a **neutral/informational badge chip** (label + tooltip/expandable body) — never error-red, never destructive, never a "fail" state. They are disclosures, not verdicts.

| Flag | Chip label | Expandable body (copy) |
|------|-----------|------------------------|
| Prop 2½ | `Prop 2½` | Prop 2½ caps the town's total **levy**, not your individual tax bill — your assessment can still rise. |
| Betterment | `Betterment` | This town may impose betterment assessments (special charges for public improvements) on top of regular property tax. |
| Title 5 | `Title 5 septic` | Properties on septic must meet Title 5 at transfer; failed systems can require a costly replacement. |
| 40B | `40B` | Chapter 40B affordable-housing developments may be present; review deed restrictions and any income/resale limits. |

Chip styling: neutral surface, informational tone; flags are additive metadata on a town and do not alter its bucket color.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | "Score towns for this budget" (the budget-input action that triggers bucketing) |
| Empty state heading | "Enter a budget to see which towns are realistic" |
| Empty state body | "Set a budget above. Towns will sort into Realistic, Stretch, and Fantasy, and the heatmap will show how each town scores." |
| Error state | "Couldn't load the town data. The MA town table failed to load — retry, and if it persists the seed data may be missing or malformed." |
| Missing-metric cell | "No data" — shown in the hatched-gray cell with the metric name; tooltip: "This metric is missing for this town and was excluded from the score (not counted as zero)." |
| Destructive confirmation | None — Phase 5 has **no destructive actions** (scoring is a pure read-only recompute over seed data; nothing is deleted or overwritten). |

Bucket labels are fixed verbatim: **Realistic / Stretch / Fantasy**, each defined relative to the entered budget (Realistic = at/below budget, Stretch = above budget up to the stretch ceiling, Fantasy = beyond it).
