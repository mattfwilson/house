# Phase 7 — UI Design Contract

> Visual and interaction contract for the thin Next.js shell. Generated by gsd-ui-researcher, verified by gsd-ui-checker.

---

## Scope Notice (read first)

Phase 7 builds **`apps/web`** — the **first** app package (it does not exist in the repo yet). It is a **thin Next.js 16 / React 19 shell** over the already-proven `packages/core` (pure calc engine) and `packages/app` (imperative shell / DI container). It holds **zero financial logic**: every number on screen comes from a core entry point (`compareScenarios`, `evaluateScenario`, `fiImpact`/`projectFiDate`, `tornado`, `scoreTowns`) reached through a thin Server Action wrapping a `packages/app` service. No money math, no bare-`number` dollars, no schema duplication in components.

This contract resolves the **app-chrome decisions the 05-UI-SPEC explicitly deferred to Phase 7** (shadcn init, component library, icon library, font, spacing scale, typography roles, color roles, navigation). It **inherits verbatim** the load-bearing color/encoding/copy contract from `05-town-scoring-heatmap/05-UI-SPEC.md` (bucket palette, heatmap encoding, MA-flag chips, heatmap copy) — those are NOT re-decided here, only referenced and extended.

**Mental model (the design brief):** a *flight simulator for a house purchase* — fly the scenario, watch the instruments (FI date, net-worth trajectory, liquidity) respond before committing. The cockpit IA (D-01/D-03), live debounced recompute (D-08), and the persistent assumptions rail (D-10) exist to make that literal.

**Hard product constraint — anti-funnel:** the UI leads with FI-impact / true affordability; bank affordability appears only as "the gap," framed as caution, never as headroom or a target. The rent-and-invest baseline is a first-class, pinned, ranking row that can win — and when it wins, that *is* the "don't buy" signal (D-04/D-05/D-06). No success-green, no aspirational "buy this" styling anywhere.

**Out of scope (unchanged):** live listings UI, auth/multi-tenant, any new financial math in the UI, live tax refresh, visx (deferred — D-13).

## Spacing Scale

8-point scale, applied **densely** (instrument-panel aesthetic, D-12 — favor the smaller tokens; the cockpit is data-forward, not airy).

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Heatmap cell gap, chip inner padding, icon-to-label gap, dense table cell padding |
| sm | 8px | Default gap inside instrument cards, between rail knob rows, table cell padding |
| md | 16px | Card padding, gap between instruments, form field spacing |
| lg | 24px | Section padding, gap between cockpit panels |
| xl | 32px | Route-level content gaps |
| 2xl | 48px | Major section breaks (rare in this dense layout) |
| 3xl | 64px | Reserved; page-level spacing |

Exceptions:
- **44px** minimum hit target for icon-only controls (delete, expand/collapse row, toggle) — accessibility floor, overrides the dense default.
- **Heatmap cells** use a fixed dense cell size (recommend 40–48px square) on a 4px (`xs`) grid gap; this is a deliberate matrix-density exception, not free-form spacing.

## Typography

Exactly 4 sizes, 2 weights. Dense and data-forward; the largest size is reserved for the one hero metric per expanded scenario (the FI-date delta).

| Role | Size | Weight | Line Height | Usage |
|------|------|--------|-------------|-------|
| Display | 28px | 600 | 1.2 | Hero FI-date delta of the expanded scenario (e.g. "+3 yr 4 mo later") — Geist Mono, tabular |
| Heading | 20px | 600 | 1.2 | Route/section titles ("Cockpit", "Town heatmap", "Sensitivity") |
| Body | 14px | 400 | 1.5 | Default UI text, form inputs, descriptions, instrument labels |
| Label | 12px | 600 | 1.4 | Table column headers, axis/legend labels, rail knob labels, chip text — dense uppercase-optional |

Weights: **400 (regular)** and **600 (semibold)** only. All dollar amounts, ratios, FI dates, and table numerics render in **Geist Mono with `tabular-nums`**, regardless of role, so figures align vertically in the comparison table and rail.

## Color

Dark instrument-panel base (D-12). The 60/30/10 split below defines app chrome; the **affordability-semantic palette is inherited verbatim from the 05-UI-SPEC** and is layered on top (it is NOT the 10% accent budget — bucket colors are data encoding, not chrome).

### 60 / 30 / 10 chrome

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#0F172A` (slate-900) | App background, cockpit surface, route canvases |
| Secondary (30%) | `#1E293B` (slate-800) | Cards/instruments, header bar (profile + scenario switcher), assumptions rail, table rows |
| Accent (10%) | `#0F766E` (teal) | See reserved-for list below |
| Destructive | `#DC2626` (red-600) | **Delete-scenario / delete-profile confirmation only** — a removal verb, never an affordability signal |

Text on dark: `#F8FAFC` (slate-50) primary text, `#94A3B8` (slate-400) muted labels/secondary readouts, `#334155` (slate-700) borders/grid lines.

**Accent (teal) reserved for:** primary CTA buttons, the **active/selected** scenario row highlight in the comparison table, the **budget / active-control** affordances (incl. the budget input driving heatmap bucketing), and the **Realistic** heatmap bucket fill. Nothing else. Not all interactive elements — secondary buttons are slate, links are slate-50 underline.

### Inherited affordability-semantic palette (from 05-UI-SPEC — render verbatim)

| Semantic | Value | Meaning / anti-funnel note |
|----------|-------|----------------------------|
| Realistic bucket | `#0F766E` teal | "within reach" — NOT success-green, NOT "you should buy" |
| Stretch bucket / **caution** | `#B45309` amber | honest "above comfort"; **also** the color of the bank-affordability gap warning (D-06) and FI-date **delay** emphasis |
| Fantasy bucket | `#64748B` slate | out of reach, muted/de-emphasized; never red-as-error |
| No-data | `#94A3B8` hatched gray | explicit "no data" marker, never a silent 0/blank/imputed cell |

### Color-honesty rules (anti-funnel, load-bearing)

- **FI-date delta (D-04) is color-honest, NOT green-good.** A *delay* (buying pushes FI later) reads in **amber** `#B45309` emphasis; *earlier* reads in neutral `#F8FAFC` / muted-teal text — **never success-green**. The hero number's job is honest disclosure, not celebration.
- **Bank affordability is amber caution, never teal/green.** The "a bank would approve ~$X more than your FI plan can absorb" readout (D-06) is rendered as a warning, never as headroom, never as a target.
- **Rent-and-invest baseline row** is visually distinct (pinned, subtle teal left-border or slate-800 emphasis) but uses **no success styling** — when it ranks #1 the ranking itself communicates "don't buy."
- **MA-flag chips** are neutral/informational badges (slate surface) — never destructive, never accent.

## Key Interaction Contracts

These are interaction (not just visual) contracts the executor must honor; they trace directly to CONTEXT.md decisions and ROADMAP success criteria.

| Contract | Requirement | Source |
|----------|-------------|--------|
| Cockpit IA | Hybrid: a home **cockpit** route + dedicated **heatmap** and **sensitivity** routes; persistent header with profile + scenario switcher; heatmap/sensitivity inherit the active profile/scenario context | D-01, D-02 |
| Cockpit = comparison | Landing shows the **ranked-by-FI-date** comparison table (all scenarios + the pinned rent baseline). Selecting a row **expands** that scenario's FI instruments + assumptions inline — comparison and "flying one scenario" are the same screen | D-03 |
| Ranking | Table ranked by **FI-date delta vs the no-purchase baseline**; net-worth-at-horizon and monthly housing delta are secondary readouts | D-04 |
| Rent baseline | A pinned, visually distinct row that also competes in the ranking; if it out-ranks every buy it sorts to #1 = the "don't buy" signal (no separate verdict chip) | D-05 |
| Hero chart | Expanded scenario shows the **trajectory-vs-baseline** net-worth line chart (Recharts) with the FI-threshold line and FI-date crossover markers on both lines; the divergence IS the cost of the house | D-07 |
| Live recompute | Editing an assumption **re-flies the instruments live, debounced ~300ms, no Apply button**; the synchronous pure core makes the Server Action round-trip cheap | D-08 |
| Working set + freeze | One **shared working assumption set** drives the whole comparison (apples-to-apples). **Save freezes** it into the scenario's snapshot; **re-opening** a saved scenario loads its frozen set as the new working set | D-09 |
| Assumptions rail | Assumptions live in a **persistent docked rail/sidebar**, always visible in the cockpit and echoed on heatmap + sensitivity routes, knobs inline next to the instruments | D-10 |
| Scenario builder | **Manual entry default**; town from the curated 24-town table; **optional** "prefill from a sample listing" (MockListingsProvider) — listings are never the required entry point | D-14 |
| Inline editor | Add/Edit is an **inline expanding editor row** within the comparison table (not a modal, not a separate route); on save it re-flies and re-ranks immediately | D-15 |
| Validation boundary | Form inputs (plain numerics → canonical decimal strings) are validated **only** through the existing core Zod schemas (`parseScenarioInputs`/`ScenarioInputsSchema`, `parseHousehold`, `parseAssumptionSet`) at the Server Action boundary; field errors surfaced from the parse result; **no duplicated validation, no bare-`number` money** in components | D-16 |
| Heatmap render | CSS-grid towns×metrics table-heatmap; cells colored by the locked bucket palette + lightness intensity (normalized composite scalar), hatched no-data cells, per-metric breakdown in tooltips — honors the 05-UI-SPEC encoding contract exactly | D-13 |
