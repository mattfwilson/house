Build the scenario + profile Server-Action layer: thin `'use server'` wrappers that (1) validate raw client input through the existing core Zod schemas (D-16), (2) call exactly ONE `@house/core` entry point or `@house/app` service, and (3) map every `Money`/`Decimal` field to strings via a dedicated DTO mapper before returning a plain serializable object. The DTO mapper is simultaneously the RSC-serialization fix and the single "no float coercion" enforcement point — class instances cannot cross the React server→client boundary.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client → Server Action | Server Actions compile to public POST endpoints; client-supplied payload is untrusted |
| Server Action → client (RSC) | Only plain data may cross back; class instances throw/strip |
| SQLite snapshot → engine | A stored snapshot blob is re-validated on load |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-7-01 | Tampering / DoS | recompare/evaluate/save actions | mitigate | Every action parses raw input through parseScenarioInputs/parseHousehold/parseAssumptionSet (D-16, ASVS V5) before any engine call; rejects before compute |
| T-7-04 | Tampering (correctness) | DTO money fields | mitigate | DTO mapper converts Money→string server-side; no `Number()` in dto/actions (eslint charts-only rule); serializability test enforces no class instance crosses |
| T-7-03 | Tampering | forged/malformed snapshot on load | mitigate | Inherited from @house/app: the SQLite adapter re-validates every snapshot leaf through parse* on load (Phase-6 T-06-12); action does not mutate snapshots |
</threat_model>
