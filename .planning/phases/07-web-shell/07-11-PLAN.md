<objective>
Deliver the profile editor — the missing entry path for the household data the entire engine runs on. Build a `ProfileEditor` form that renders all nine `Household` leaf fields (plus the profile name) for input, wired to `saveProfileAction`/`deleteProfileAction`, with field-level errors surfaced from the core `parseProfile`/`parseHousehold` Zod parse (D-16). Expose it on a dedicated `/profile` route that supports creating the FIRST profile from an empty state and editing/deleting existing ones, and add the entry affordance to the persistent header.

Purpose: SC-1 ("User can edit profiles, assumptions, and scenarios through forms") and 07-CONTEXT.md's domain boundary both list profile forms as a deliverable, and PROF-01/02/03 are undelivered in the UI tier without this. 07-03 created the profile Server Actions + `toProfileDTO` and 07-05's Header has a profile *switcher* (a picker among existing profiles), but nothing renders the Household fields for user *input* — so a brand-new user cannot create the first profile that drives the affordability/FI engines. This plan closes that gap.
Output: `profile-form.ts` (+ test), `ProfileEditor.tsx`, the `/profile` route, and the header entry affordance.
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| profile form → Server Action | Name + nine household leaves cross to the public POST saveProfileAction; input is untrusted |
| Server Action → client (RSC) | Only plain DTO data (decimal strings) may cross back to pre-fill the editor |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-7-01 | Tampering / DoS | profile editor inputs | mitigate | All input parsed through parseProfile/parseHousehold at saveProfileAction (D-16, ASVS V5); the editor holds no schema and surfaces field errors from the core parse result |
| T-7-04 | Tampering (correctness) | bare-number money in the form | mitigate | formToRawProfile converts every numeric to a canonical decimal string at the edge; test asserts every leaf is `typeof === 'string'`; no `Number()` math in the editor |
| T-7-08 | Tampering | client re-checking / bypassing the ≤2 cap | mitigate | The cap stays a profile-service invariant (saveProfile); the UI shows MAX_PROFILES for copy only and performs no enforcement |
</threat_model>
