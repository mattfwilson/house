<objective>
Close the single verification gap (CR-01 + CR-02 from the code review): both binary-search
max-price solvers â€” `bankAffordability`'s inline solver and the shared `solveMaxPrice` in
`true-affordability.ts` â€” assume the bisection precondition `passes(low)` holds and proceed to
bisect even when it is false. For an infeasible household (e.g. `availableNetWorth âˆ’ reserve <
downPaymentCash + closingCosts(low)`, or a `targetSavingsRate` unmet at every price), the solver
silently returns a plausible-but-wrong ceiling (â‰ˆ`downPaymentCash + 1`) instead of $0/infeasible.
Secondary defect (CR-02): if the bracket-doubling cap is exhausted while `passes(high)` is still
true, bisection returns a non-maximal ceiling silently.

The fix is two guards in each solver plus tests, and nothing else. **Decision (resolving the
"$0 vs feasible:false" choice in the gap):** return `Money.zero()` from the solver when
`!passes(low0)`. Rationale â€” this keeps ALL FOUR affordability result type SHAPES unchanged
(`BankAffordabilityResult`, `TrueAffordabilityResult`, `AffordabilityGapResult`,
`EvaluateScenarioResult` are untouched), so `affordability.type-test.ts` needs no new guards,
`index.ts` exports no new symbols, and `gap.ts` requires no change: a $0 ceiling composes
acceptably through the existing `min`/`signedGap`/`toCents()` verdict logic (the gap simply
reports a $0 true or bank ceiling, which is the honest "infeasible at this profile" answer). No
`feasible:false` field is added in this gap-closure scope. The bracket-cap exhaustion (CR-02) is
converted to a thrown Error â€” a loud, diagnosable failure consistent with "trustworthiness is the
product."

Purpose: an infeasible household must NOT receive a fundable-looking small price ceiling â€” that is
the exact "trustworthiness is the product" failure CLAUDE.md warns against (D-15 / CORE-02).
Output: two source files guarded, two test files extended; feasible-household behavior and the
golden snapshot provably unchanged.
